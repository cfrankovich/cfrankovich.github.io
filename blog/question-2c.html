intro.

Question 2C can be very daunting to even look at. I'm willing to bet most people would skip it upon seeing how much is given. However, this question only tests you on your direction following skills an a bit of common sense. If this web page doesn't teach you everything about Question 2C, please contact me and I'll be happy to help. Let's begin by digesting the problem.

the question.

The first part of the question throws weird words at you that you don't need to know. All it's doing is restating what they give you. The second part of the question is the task. A trace table is essentially a tool programmers will use to debug their code in an organized way. The trace table is given to us so all we have to do is debug an assembly program.

given.

We are given a number of confusing things but this section will break down each one. First off, we are given the instruction set. This table is our instructions to follow when filling out the trace table. You could also think of this like memory in a way where address 20 contains the instruction "LDD 96." More on this later. The second part of this table are addresses 93 to 98. These just hold the values next to them in memory (you'll see on the trace table). And the third part is the IX which is 8. All of this information is crucial to setting up our trace table later on.

The second thing we're given is the ASCII code table. This isn't very relevant until we reach the certain instruction. However, if you look at the memory, you'll notice 453 comes up often. Could these be related?

The third given table is the most important and allows us to solve the problem. This is the table that holds all instructions that is directly related to the instructions in the first given table. Op code means the operation code which is the abbriviation for an instruction. Operand is basically the argument to the instruction and is typically used to perform an operation. The explanation explains what the instruction is doing and how to fill it out in the trace table.

The final given table is the trace table itself. This looks scary and confusing but it just acts as a template for us to fill in and debug our instructions. I will go more in depth about the trace table.

trace table.

Earlier, you saw that the trace table looked like some giant spreadsheet or something of the sort. Let's go to the top and explore what the trace table is. The trace table is built in a way to make each row one instruction line. This will make more sense once we begin. Five main parts of the trace table:

  • Instruction Addresses
  • ACC
  • Memory Address
  • IX
  • OUTPUT
The instruction addresses should go from 20 to 32 since these are simply indicating which instruction is being processed on that row of the table. The ACC is just some place to store data. Memory address are the different memory addresses (93-98) and are represented by the whole column. IX is kinda like the ACC its just a place to store data. However, we put 8 here because there was an instruction in the first table that had the IX as 8. And output is the output of the program.

explanation.
